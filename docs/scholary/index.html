1. Abstract

Nobel Prize Explorer is a web application that integrates with the official Nobel Prize SPARQL endpoint to retrieve Nobel prizes, laureates and their relationships. The platform provides a responsive UI for exploration (filters + pagination), a statistics area based on SPARQL aggregations, and user-specific favorites stored as RDF (ActivityStreams Like) in a local Turtle file.

The backend is implemented in Python (Flask) and exposes a REST API that abstracts SPARQL complexity and returns JSON. Authentication relies on Flask-Login session cookies, with email confirmation via Mailtrap (testing inbox).

2. System Architecture (SOA + C4)

2.1 High-level SOA

  • Frontend (React + Tailwind + Zustand): pages for prizes, laureates, statistics, favorites, authentication.
  • Backend (Flask): REST API endpoints, session-based auth, SPARQL client, RDF local store.
  • External endpoints: Nobel Prize SPARQL endpoint (core data source).
  • Persistence: SQLite (users) + Turtle file (RDF likes).

3. Internal Data Models

3.1 SQL (SQLite via SQLAlchemy)

The relational store persists application users and email-confirmation state. Core entity: User with (id, mail, password hash, confirmed flag).

3.2 RDF Local Store (Turtle)

Favorites are stored as RDF nodes of type as:Like, linking the actor (user URI) to an object (e.g., a Nobel prize URI), together with timestamps (schema:dateCreated) and an internal kind marker (nexp:kind).

4. API Design (REST + OpenAPI)

4.1 REST Endpoints Overview

  • /auth/register, /auth/login, /auth/logout, /auth/check, /auth/delete
  • /api/nobel (list prizes with limit/offset/category)
  • /api/nobel/count (count prizes for pagination)
  • /api/nobel/stats (aggregations; optional yearFrom/yearTo)
  • /api/laureates and /api/laureates/<id>
  • /api/rdf/likes (GET/POST/DELETE, requires session cookie)

5. RDF Knowledge Model

The application reuses established vocabularies such as ActivityStreams (as:), Schema.org (schema:), FOAF (foaf:) and XSD. This ensures interoperability and enables favorites to be queried as a graph, not only as relational rows.

RDF “Like” pattern: each saved item is represented as an as:Like node with: as:actor (user URI), as:object (saved Nobel resource URI), schema:dateCreated (timestamp), nexp:kind (internal type marker).

6. External Knowledge Sources + SPARQL

Nobel data is retrieved through SPARQL queries against the Nobel Prize endpoint. The backend implements pagination-friendly patterns (LIMIT/OFFSET + a second VALUES-based detail query) to keep responses bounded and stable.

7. Linked Data Principles

  • URIs: Nobel entities are identified by stable URIs from the public dataset.
  • Reuse of vocabularies: ActivityStreams + Schema.org + FOAF ensure semantic interoperability.
  • RDF serialization: user actions are stored in Turtle (non-proprietary, machine-readable).
  • Linkability: favorites reference external URIs directly, enabling future enrichment and federated exploration.

8.1 UC-01 — Register Account

Goal: Create a new user account and initiate email verification.

Primary Actor: User

Preconditions: The email is not already registered.

Trigger: User submits the Register form.

Main Success Scenario

  1. User opens the Register page and enters email + password.
  2. Frontend validates basic input (format/required fields) and sends a register request.
  3. Backend hashes the password and stores the user with confirmed=false.
  4. Backend generates a confirmation token and sends a confirmation email via Mailtrap SMTP.
  5. Frontend displays the “registration success” page with instructions.

Alternative / Error Flows

  • A1: Email already exists → backend returns an error; frontend displays a message.
  • A2: Mail sending fails → account may exist but user is not confirmable; frontend displays retry instructions.

API + Data Impact

  • Endpoint: POST /auth/register
  • Data written: SQLite (User row)
  • External system: Mailtrap SMTP (email delivery in testing inbox)

8.2 UC-02 — Confirm Email

Goal: Activate the account by validating the emailed confirmation token.

Primary Actor: User

Preconditions: User has a pending (unconfirmed) account and can access the Mailtrap inbox.

Trigger: User clicks the “Confirm email” link received by email.

Main Success Scenario

  1. User opens Mailtrap and locates the confirmation email.
  2. User clicks the confirmation link which calls the backend confirmation route.
  3. Backend validates token integrity and expiry, then updates user state to confirmed.
  4. Frontend shows an “Email confirmed successfully” screen.

Alternative / Error Flows

  • A1: Token expired → user must re-register or request a new token (if implemented).
  • A2: Token invalid/tampered → backend rejects confirmation.

API + Data Impact

  • Endpoint: GET /auth/confirm/<token> (or similar)
  • Data updated: SQLite user row (confirmed=true)

8.3 UC-04 — Browse Prizes with Filters + Pagination

Goal: Explore Nobel prizes efficiently via category filter + bounded pagination.

Primary Actor: User

Preconditions: Backend can access the Nobel SPARQL endpoint.

Trigger: User opens the Prizes page or applies filters.

Main Success Scenario

  1. User opens the Prizes page.
  2. User optionally selects a category and a page size (limit).
  3. Frontend requests a page of prizes from the backend.
  4. Backend issues SPARQL queries using a pagination strategy (LIMIT/OFFSET).
  5. Backend returns JSON; frontend renders responsive cards and pagination controls.

Implementation Detail (Performance Pattern)

  • Use a first SPARQL query to fetch only prize URIs for the current page.
  • Use a second SPARQL query with VALUES to fetch details for those URIs.
  • Use a separate COUNT(DISTINCT ?prize) query to compute total results.

API + Data Impact

  • Endpoints: GET /api/nobel, GET /api/nobel/count
  • External system: Nobel Prize SPARQL endpoint (read-only)

8.4 UC-06 — Save / Remove Favorites (RDF Like)

Goal: Persist user favorites as Linked Data (ActivityStreams Like) and manage them (list/remove).

Primary Actor: User

Preconditions: User is authenticated (valid session cookie).

Trigger: User clicks Save on a prize card or Remove on favorites.

Main Success Scenario — Save

  1. User clicks Save on a prize card.
  2. Frontend sends an authenticated request to the Likes endpoint.
  3. Backend creates an RDF node of type as:Like.
  4. Backend writes triples: as:actor, as:object, schema:dateCreated, nexp:kind.
  5. Backend persists the graph to data/local.ttl.

Main Success Scenario — List/Remove

  1. User opens Favorites page → frontend requests likes for the current user.
  2. Backend loads Turtle graph and filters likes by actor (user URI).
  3. User clicks Remove → backend deletes triples for the matching Like node and saves the file.

API + Data Impact

  • Endpoints: GET /api/rdf/likes, POST /api/rdf/likes, DELETE /api/rdf/likes
  • Data written: Turtle file (data/local.ttl)

8.5 UC-07 — Statistics Dashboard (SPARQL Aggregations)

Goal: Provide aggregated insights over Nobel data using server-side SPARQL aggregation queries.

Primary Actor: User

Preconditions: SPARQL endpoint is reachable; category/year filters are optional.

Trigger: User opens Statistics page or applies filters.

Main Success Scenario

  1. User opens the Statistics dashboard.
  2. User optionally selects a category and a year interval.
  3. Frontend requests statistics from the backend.
  4. Backend computes totals and breakdowns using SPARQL (COUNT, GROUP BY).
  5. Frontend renders charts and summary numbers responsively.

API + Data Impact

  • Endpoint: GET /api/nobel/stats
  • External system: Nobel Prize SPARQL endpoint (read-only)
